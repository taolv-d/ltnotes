# 计算机体系结构

1. 计算机发展至今仍然遵循冯诺伊曼结构，即存储程序和指令执行。虽然冯诺伊曼结构所有东西都保存在内存中，很容易让访存成为性能瓶颈。

2. CPU 的现状（功耗、性能、成本）

   1. 指令体系确定后，计算机结构设计的重点是降低每条指令执行的周期数（快速执行单个指令），或者提高每个周期执行的指令数（指令级并行）

   2. 主频宏观上取决于为结构设计，微观上取决于工艺和电路设计（宏观上，最基本的指令如加法的门延迟决定了一个周期的最短时间，微观上，半导体的性能、功耗等又限制了主频）

   3. CPU 跟操作系统厂家一起来让用户在3～5年更换电脑，手段包括控制芯片老化寿命、更新软件不兼容旧版本（操作系统、文档格式等）、推出需要更高性能的软件

   4. 半导体工艺升级是降低动态功耗的有效方法，他通过降低电容电压，成倍的降低功耗

   5. 功耗跟电压成正比，跟频率的平方成正比

   6. 当前 1bit 翻转需要的功耗是理论的10^10倍

   7. CPU 中用于进行加减乘除运算的面积只占总面积的很小一部分，大部分面积用来保存指令和数据（高速缓存），80%的晶体管用在了cache上

   8. 由于CPU有很多引脚，其封装成本很高，甚至有些已经超过硅部分的成本了

3. 体系结构设计的基本原则

   1. 平衡性、局部性、并行性、虚拟化

   2. 指令级并行提升的空间已经很小（原因是1. 平均每N个指令就会有一次跳转指令，同时分支预测也无法保证100%准确性，这里限制了进一步的提升 2. Cache 缺失等引入的延迟也会限制并行度）

# 指令系统

* 指令系统的设计原则

  1. 兼容性、通用性、高效性、安全性

  2. 体系结构的发展与指令系统兼容性的基本原则要求是矛盾的，为了兼容一定会背上历史的包袱

  3. RISC指令集最本质的特征是通过 load store 结构简化了指令之间的关系，所有指令都是对寄存器操作的

  4. 除X86系统外，当前的体系结构基本都是寄存器-寄存器型（即源操作数、目标操作数都是寄存器）。寄存器-寄存器类型的优势是：

     1. 寄存器访问速度快，便于编译器优化、可以充分利用程序的局部性原理，大部分操作可以在寄存器中完成

     2. 寄存器-寄存器容易判断寄存器之间的相关性，容易实现流水线、多发射、乱序执行等技术

* 指令系统组成

  1. 最常见的寻址方式：偏移量寻址、立即数寻址、寄存器间接寻址

  2. 程序中的switch、函数指针、虚函数调用和过程返回都属于间接转移（cache 亲和性差？）

  3. 寄存器窗口：有几组寄存器，函数调用时不用保存寄存器现场，只需要切换一组没有使用的寄存器给子函数用就行（这样是不是有点浪费资源？毕竟寄存器很宝贵）

* 特权指令

  1. 异常与中断

  2. 存储管理

     1. 隐藏与保护：可以方程用户程序随意修改自己的代码段（例如电脑病毒）

     2. 为每个程序分配连续的地址空间（且不冲突，实际物理地址是不连续的）

     3. 扩展物理地址（运行超过内存大小的程序，程序的局部性原则，实际运行时不需要全部加载到内存中）

     4. 节约物理内存：没用到部分先不加载到内存

     5. TLB（转换后援缓冲器，页表缓存 快表）：利用程序的局部性原理，存储当前处理器中最经常访问的页表。处理器用地址空间标识符（Address Space Identifier ASID）和虚拟页号（virtual Page Number， VPN）在TLB中进行匹配。TLB 与 cache 类似，可以当作页表的cache

     6. TLB 缺失异常时如果加载的页表也缺失怎么办？

        1. 如果TLB 缺失，加载TLB 的过程中又出现缺失异常，此时就会出现中断嵌套，除非有两套硬件，否则无法支持

        2. 因硬件上只维护了这一套保存上下文专用的寄存器，所以需要确保在TLB 重填异常处理过程中不再触发TLB 重填异常

        3. 处理器因TLB 重填异常触发而陷入异常处理后，硬件会自动将虚实地址翻译模式调整为直接地址翻译模式，从而确保TLB 重填异常处理程序第一条指令的取指和访存8一定不会触发TLB 重填异常

        4. 软件设计人员（操作系统）也要保证后续TLB 重填异常处理返回前的所有指令的执行不会触发TLB 重填异常。

     7. Malloc 分配内存时一般会产生两次异常？

        1. 一次是TLB 缺失（此时会建立虚拟地址到物理地址的对应）

        2. 第二次是对应的内存块缺失，此时会加载内存块到缓存。

        3. **产生两次异常而非一次完成所有操作的原因是保证TLB 重填异常的处理速度。**

     8. Cache 的每一个单元不仅要存储数据，还要存储该数据对应的内存地址（称为Cache 标签，Tag）以及在Cache 中的状态（如是否有效，是否被改写等）。

  3. 同步机制

     1. 当确认竞争者在**另一个CPU 核上**，而且临界区程序很短时，让等待锁的线程循环检查锁状态直至锁可用显然是合理的，这也是Linux 为SMP （Symmetric Multi‑Processing）实现的自旋锁。

     2. 但当竞争者都在**同一个CPU 核上时**，在不可抢占的内核下进行自旋可能导致死锁，此时自旋锁将退化为空操作。

     3. 当自旋锁不可用时，需要使用互斥锁的机制。当一个线程获取锁失败时，会将自己阻塞并调用操作系统的调度器。在释放锁的时候还需要同时让其他等待锁的线程离开阻塞状态。挂起和唤醒线程的操作与指令系统无关，但测试锁状态和设置锁的代码依赖于原子的“测试并设置” 指令，而LoongArch 指令系统的实现方式是LL/SC 指令（对32 位操作加.W 后缀，64 位加.D 后缀）。LL 指令设置LL bit，并检测访问的物理地址是否被修改或可能被修改，在检测到时将LL bit 清除。在SMP 中，检测LL bit 通常使用Cache 一致性协议的监听逻辑来实现。在单处理器系统中，异常处理会破坏LL bit。SC 指令实现带条件的存储。当LL bit 为0 时，SC 不会完成存储操作，而是把保存值的源操作数寄存器清零以指示失败。

  4. 事务内存：核心思想是通过尝试性地执行事务代码，在程序运行过程中动态检测事务间的冲突，并根据冲突检测结果提交或取消事务。

* 计算机组成原理和结构

  1. 冯诺伊曼结构的改进（内存瓶颈）：

     1. 由以运算器为中心改进为以**存储器为中心**。使数据的流向更加合理，从而使运算器、存储器和输入输出设备能够并行工作。

     2. 由单一的集中控制改进为分散控制。计算机发展初期，工作速度很低，运算器、存储器、控制器和输入输出设备可以在同一个时钟信号的控制下同步工作。现在运算器、内存与输入输出设备的速度差异很大，需要采用异步方式分散控制。

     3. 从基于串行算法改进为适应并行算法。出现了流水线处理器、超标量处理器、向量处理器、多核处理器、对称多处理器（SymmetricMultiprocessor，简称SMP）、大规模并行处理机（Massively Parallel Processing，简称MPP）和机群系统

     4. 出现为适应特殊需要的专用计算机，如GPU、DSP。

     5. 在非冯·诺依曼计算机的研究方面也取得一些成果，如依靠数据驱动的数据流计算机、图归约计算机等。

  2. 哈佛结构（指令和数据分开存储），主要在嵌入式领域（mcu）

  3. 控制器的优化策略：

     1. 转移预测：冯诺伊曼体系结构的**指令驱动**特点造成**转移指令**成为流水线设计的难点

     2. 转移预测技术：利用for循环这种代码逻辑的特点，对于一个转移指令的表现可能与历史状态，或者附近的其他转移指令的结果有关，因此可以结合这些信息推测，并以推测结果执行。一旦预测错，只能丢弃当前计算结果，重新执行正确的指令

     3. 乱序执行技术：cache 未命中时，可以继续执行已经加载的其他指令，存在指令冲突时，也可以先执行后面不冲突的指令。

        1. **译码读寄存器阶段判断指令能否执行**（关注依赖的寄存器是否已经更新）：如果操作数已经准备好，就进入执行阶段；如果操作数没有准备好，就进入称为保留站或者发射队列的队列中等待，直到操作数准备好后再进入执行阶段。

        2. 如何保证程序有序结束（精确异常需求）：执行完的指令均进入一个称为重排序缓冲（Re‑Order Buffer，简称ROB）的队列，并把执行结果临时写入重命名寄存器。ROB 根据指令进入流水线的次序，有序提交指令的执行结果到目标寄存器或存储器。

        3. 记分板+保留站最早实现了指令的动态调度。

        4. 保留站、重排序缓冲、重命名寄存器都是微结构中的数据结构，程序员无法用指令来访问，是结构设计人员为了提高流水线效率而用来临时存储指令和数据的。其中，保留站把指令从有序变为无序以提高执行效率，重排序缓存把指令从无序重新变为有序以保证正确性，重命名寄存器则在乱序执行过程中临时存储数据。

        5. **重命名寄存器**（不同指令都依赖同一个寄存器，但没有数据相关）：两组执行不同运算但使用同一结构寄存器的指令可以使用不同的重命名寄存器，从而避免该结构寄存器成为串行化瓶颈，实现并行执行。

     4. 超标量技术（多套硬件）：超标量结构允许指令流水线的每一阶段同时处理多条指令。由于超标量结构的指令和数据通路都变宽了，使得寄存器端口、保留站端口、ROB 端口、功能部件数都需要增加。现代超标量处理器一般包含两个以上访存部件，两个以上定点运算部件以及两个以上浮点运算部件。超标量结构在指令译码或寄存器重命名时不仅要判断前后拍指令的数据相关，还需要判断同一拍中多条指令间的数据相关。

* 计算机启动过程：

  1. 无论采用何种指令系统的处理器，复位后的第一条指令都会从一个预先定义的特定地址取回。处理器的执行就从这条指令开始（这里就是复位程序，那么复位过程中应该要屏蔽中断吧？）

  2. 所谓初始化，实际上是将计算机内部的各种寄存器状态从不确定设置为确定，将一些模块状态从无序强制为有序的过程（执行一系列load store 指令）

  3. 初始化第一步：从BIOS 取指令

     1. 处理器的第一条指令实际上是由复位信号控制的，但受限于各种其他因素，复位信号并没有对处理器内部的所有部分进行控制，例如TLB、Cache 等复杂结构，而是只保证从取指部件到BIOS取指令的通路畅通。

     2. 处理器复位后先是通过频率为几十兆赫兹（MHz）以下的低速设备取指令，例如SPI 或LPC 等接口。一拍只能取出1 比特（SPI）或4 比特（LPC），而一条指令一般需要32 比特。对于吉赫兹（GHz）的高性能处理器来说，几千拍才能执行一条指令，

  4. 调试接口初始化

  5. TLB 初始化

  6. Cache 初始化

  7. 复位完成后处理器处于最高特权状态中，完成各种初始化

  8. Cache 能够加速，为什么不先初始化cache？在跳转到Cache 空间执行后，程序运行效率大大提升，随之而来的是处理器内各种复杂猜测机制的使用。例如对取数操作的猜测执行可能导致一个落在TLB 映射空间的访存操作，如果此时TLB 尚未完成初始化，就可能会导致TLB 异常的发生，而TLB 异常处理机制的缺失又会导致系统的崩溃。实际上，在跳转到Cache 空间执行前，BIOS 中还会对一些处理器具体实现中的细节或功能进行初始化，在保证执行基本安全的状态下，才会跳转到Cache 空间执行。这些初始化包括对各种地址窗口的设置、对一些特殊寄存器的初始化等。

  9. 多核系统初始化：

     1. 分为主核跟从核

     2. 初始化阶段使用信箱机制通信（简单理解就是一个大的状态寄存器，每一位代表不同含义，通常仅在初始化时使用，原因容量带宽有限）。

     3. 从核完成自身初始化后会等待主核唤醒（信箱）。

     4. 有时主核也可以给从核安排初始化任务（例如初始化TLB、cache等，加速启动过程）

     5. 为什么Linux 内核中的核间中断处理不通过信箱寄存器进行呢？首先信箱寄存器只有一组，也就是说如果通过信箱寄存器发送通信消息，在这个消息没被处理之前，是不能有其他核再向其发出新的核间中断的。这样无疑会导致核间中断发送方的阻塞。另外，核间中断寄存器实际上是IO寄存器，前面我们提到，对于IO 寄存器的访问是通过不经缓存这种严格访问序的方式进行的，相比于Cache 访问方式，不经缓存读写效率极其低下，本身延迟开销很大，还可能会导致流水线的停顿。因此在实际的内核中，只有类似休眠唤醒这种特定的同步操作才会利用信箱寄存器进行，其他的同步通信操作则是利用内存传递信息，并利用核间中断寄存器产生中断的方式共同完成的。

* 关于数字电路等基本记录，这里就不多展开了：

  1. 从原理上看，只要有一个二值系统，并且系统中能够进行与、或、非这样的基本操作，就能够搭建出一台计算机。

  2. 在CPU 设计中，加法器的延迟是决定其主频的一个重要参数指标，如果加法器的延迟太长，则CPU 的主频就会降低。 （因此在设计加法器、等基本器件时要考虑门延时，避免影响主频，一个例子就是加法器如果考虑进位只能一位一位算，就会有巨大的门延迟，计算机通过数字信号特性做了变换，让进位信号的计算可以并行计算出来，通过级联等手段尽可能在并行和硬件复杂度之间平衡，从而降低门延迟）

* 流水线处理器

  1. 单周期处理器模型：每个时钟周期必须完成取指、译码、读寄存器、执行、访存等很多组合逻辑工作，为了保证在下一个时钟上升沿到来之前准备好寄存器堆的写数据，需要将每个时钟周期的间隔拉长，导致处理器的主频无法提高。使用流水线技术可以提高处理器的主频。

  2. 多周期处理器：RAW （上一题条写寄存器的指令还没写回寄存器，下一条指令就要读这个寄存器的值了，结果拿到旧值） 会引起流水线冲突。（WAR WAW 不会引起流水线冲突，但是会造成乱序执行冲突，后面介绍）。解决办法就是流水线前递技术，简单说就是计算的结果不等最后写回，而是算出来马上就更新，

  3. 动态调度（这里与8重复了）：

     1. 保留站：有些指令由于依赖前面指令结果不能马上执行，需要用一个结构存放这些等待的指令，这个结构被称为保留站，有的文献中也称之为发射队列

     2. 寄存器重命名：WAR 和WAW 同RAW 是有本质区别的，它们并不是由程序中真正的数据依赖关系所引起的相关关系，而仅仅是由于恰好使用具有同一个名字的寄存器所引起的名字相关。（Tomasulo算法，在记分板技术之上改进而来）

     3. 精确异常，异常发生后的指令不能影响机器状态，异常前的指令都已经执行完

        1. 在流水线中添加一个重排序缓冲（ROB）来维护指令的有序结束

        2. 同时在流水线中增加一个“提交” 阶段。指令对机器状态的修改只有在到达提交阶段时才能生效（软件可见），处于写回阶段的指令不能真正地修改机器状态，但可以更新并维护一个临时的软件不可见的机器状态。

        3. ROB 是一个先进先出的有序队列，所有指令在译码之后按程序顺序进入队列尾部，所有执行完毕的执行从队列头部按序提交。

        4. 提交时一旦发现有指令发生异常，则ROB中该指令及其后面的指令都被清空。发生异常的指令出现在ROB 头部时，这条指令前面的指令都已经从ROB 头部退出并提交了，这些指令对于机器状态的修改都生效了；异常指令和它后面的指令都因为清空而没有提交，也就不会修改机器状态。这就满足了精确异常的要求。

  4. 总结一下实现动态调度后流水线各阶段的调整

     1. 取指, 不变

     2. 译码, 译码拆分为译码和读操作数两个阶段。在读操作数阶段，把操作队列的指令根据操作类型派送（dispatch）到保留站（如果保留站以及ROB 有空），并在ROB 中指定一项作为临时保存该指令结果之用；保留站中的操作等待其所有源操作数就绪后即可以被挑选出来发射（issue）到功能部件执行，发射过程中读寄存器的值和结果状态域，如果结果状态域指出结果寄存器已被重命名到ROB，则读ROB。

     3. 执行, 不变

     4. 写回。把结果送到结果总线，释放保留站；ROB 根据结果总线修改相应项。

     5. 提交。如果队列中第一条指令的结果已经写回且没有发生异常，把该指令的结果从ROB 写回到寄存器或存储器，释放ROB 的相应项；如果队列头的指令发生了异常或者转移指令猜测错误，清除操作队列以及ROB 等。

# 并行与多核

* 并行程序

  1. 并行级别

     2. 数据级并行：SIMD

     3. 任务级并行

  2. 并行编程模型：

     1. 单任务数据并行

     2. 多任务共享内存

* 多核：

  1. 单核优化、指令集并行等的优化已经很难进一步提升了

     1. 在达到每个时钟周期发射执行4 条指令后就很难有性能收益。

     2. 提高电压和主频的方法导致功耗随着主频的提高超线性增长。开关功耗跟主频成正比，跟电压的平方成正比，即在一定范围内功耗与主频的三次方成正比。（反过来降低主频也能降低功耗）

  2. 多核cache 设计：

     1. UCA结构 片内共享最后一级Cache（Last Level Cache，LLC），片间共享内存。这种集中式的共享LLC，很容易随着处理器核数目的增加成为瓶颈，可扩展性受限。适用核数少的情况

     2. NUCA 是一种分布式共享结构，每个处理器核拥有本地的LLC，并通过片上互连访问其他处理器核的LLC。在NUCA 结构中，处理器核可以访问所有的LLC，但是不同位置的LLC 具有不同的访问延迟。

  3. Cache 一致性：一个数据块可能在主存和Cache 之中保存多份，而不同的处理器核有可能同时读取或者修改这个数据，导致不同的处理器核观察到的数据的值是不同的。

     1. 写无效协议的优点是：一旦某处理器核使某一变量在所有其他Cache 中的备份无效后，它就取得了对此变量的独占权，随后它可以随意地更新此变量而不必告知其他处理器核，直到其他处理器核请求访问此变量而导致独占权被剥夺。

     2. 写无效缺点是：当某变量在一处理器核中的备份变无效后，此处理器核再读此变量时会引起Cache 不命中，在一个共享块被多个处理器核频繁访问的情况下会引起所谓的“乒乓” 效应，即处理器核之间频繁地互相剥夺对一个共享块的访问权而导致性能严重下降。

     3. 写更新协议的优点是：一旦某Cache 缓存了某一变量，它就一直持有此变量的最新备份，除非此变量被替换掉。

     4. 写更新协议其缺点是：写数的处理器核每次都得把所写的值传播给其他处理器核，即使其他处理器核不再使用所写的共享块。写无效协议适用于顺序共享（Sequential Sharing）的程序，即在较长时间内只有一个处理器核访问一个变量；而写更新协议适用于紧密共享（Tight Sharing）的程序，即多个处理器核在一段时间内频繁地访问同一变量。

  4. 侦听协议与目录协议：

     1. 侦听协议的基本思想是，当一个处理器核对共享变量的访问不在Cache 命中或可能引起数据不一致时，它就把这一事件广播到所有处理器核。系统中所有处理器核的Cache 都侦听广播，当拥有广播中涉及的共享变量的Cache 侦听到广播后，就采取相应的维持一致性的行动（如，使本Cache 的备份无效、向总线提供数据等）。侦听协议实现较简单，每个处理器核Cache 只需要维护状态信息就可以了。侦听协议适合于通过总线互连的多核处理器。存在可伸缩性差的问题。

     2. 在采用片上网络互连的多核处理器中通常使用基于目录的Cache 一致性协议。目录协议的主要思想是，为每一存储行维持一目录项，该目录项记录所有当前持有此行备份的处理器核号以及此行是否已被改写等信息。当一个处理器核欲往某一存储行写数且可能引起数据不一致时，它就根据目录的内容只向持有此行的备份的那些处理器核发出写使无效/写更新信号，从而避免了广播。典型的目录组织方式为位向量目录。位向量目录中的每一目录项有一个n 位的向量，其中n 是系统中处理器核的个数。位向量中第 i 位为“1” 表示此存储行在第i 个处理器核中有备份。每一目录项还有一改写位，当改写位为“1”时表示某处理器核独占并已改写此行。位向量目录的缺点是，所需的目录存储器容量随处理器核数n 以及共享存储容量m 的增加以O(m\*n) 的速度增加，有较大存储

* 互联结构：

  1. 片上总线

  2. 交叉开关

  3. 片上网络NOC。

  其中共享总线结构和交叉开关结构因可伸缩性差的原因，主要用于小规模的多核处理器；片上网络（Network‑on‑Chip，简称NOC）具有可伸缩性好的优势，适合于核数较多的多核/众核处理器。

* 多核处理器的同步：LL/SC 是最常见的原子指令，使用原子指令对LL/SC 实现原子操作方式的过程如下：

  1. 首先，LL 指令将对应地址的内存数据读入寄存器，然后可以对该寄存器中的值进行任意的运算，最后使用SC 指令尝试将运算后的数据存回内存对应的地址。

  2. 当且仅当LL 指令完成之后没有其他对该地址内存数据的修改操作，则SC 指令执行成功并返回一个非零值，运算后的数据顺利写回内存，否则SC 指令执行失败并返回值0，修改后的数据不会被写回内存，也不会产生任何对内存的改动。

  3. SC 指令失败后一般需要重新执行上述过程，直到SC 指令成功为止。

  4. SC 指令的成功说明了LL/SC 指令之间没有其他对同一地址的写入操作，也就保证了LL/SC 指令之间的不可分割性。
