4. # 缓存
    

5. 为什么CPU不直接操作内存：
    
    1. 计算机运算速度提升，但是存储系统传输数据的速度提升很小
        
    2. 存储容量越大，传输越慢（寻址越复杂）
        
    3. 速度越快的存储器成本越高
        
        |   |   |   |   |
        |---|---|---|---|
        |存储结构|容量大小|访问时间(ns)|成本|
        |L1 cache|32 ~ 64kB|1|锁存器，成本高（6T）|
        |L2 cache|256 kB ~ 1MB|4||
        |L3 cache|2~64 MB|10~20||
        |内存|4GB|50~100|电容存储，成本低|
        
6. 为什么会出现多级缓存结构？
    
    1. 由于成本和速度的限制，我们没办法设计一个又快，容量又大的L1 cache 来为CPU高速运算提供可靠的存储
        
    2. 同时由于程序具有 **时间局部性**和**空间局部性**，因此CPU同一时刻只需要操作非常集中的一小部分数据即可
        
    3. 因此我们只需要一个小的、速度跟CPU一致的memory，把CPU最常用的数据放在这里即可，即L1 cache
        
    4. 但是如果有需要的数据不在L1 cache 中，直接从内存加载速度太慢了，可能造成程序性能极具下降，为此，提供多个层级，L1 未命中从L2找，L2没有从L3找，尽可能的降低未命中带来的代价
        

显然，如何决定哪些块要放在缓存中、缓存满了如何替换、缓存块大小这些都是决定缓存效果的重要因素：

3. 六种基本缓存优化策略
    
    1. 增大块大小：利用空间局部性降低失效率，但可能增加失效开销。
        
    2. 增大缓存容量：降低失效率，但可能增加命中时间和成本。（受限功耗、频率很难再增大）
        
    3. 提高关联度：降低冲突失效率，但可能增加命中时间。
        
    4. 多级缓存：降低失效开销。
        
    5. 读失效优先于写：读 Miss 时，先处理读，再处理写缓冲中的写，减少读失效延迟。
        
    6. 避免虚实地址转换引起的开销。
        
4. 五种缓存优化思想
    
    1. 减少命中延迟
        
    2. 增加Cache带宽
        
    3. 减少缺失惩罚
        
    4. 增加命中率
        
    5. 通过并行来减少缺失惩罚或缺失率
        
5. 十种高级缓存优化策略
    
    1. **小且简单的L1缓存**（高时钟频率、功耗限制，不能提高），目的是尽可能降低访问延迟（一个周期访问）。目前主要在相联度上下文章
        
    2. **路预测**，提前预测下一次可能访问哪些块（使用1bit 来记录），避免寻址带来的时间开销，失败会带来惩罚
        
    3. cache 流水化
        
    4. **缓存访问非阻塞（提高带宽）**：缓存未命中不会阻塞CPU（等待缓存加载好再执行），而是继续尝试执行其他指令，通常可以弥补L1 数据缺失 L2数据缺失带来的惩罚，但更低级的缺失无法补偿
        
    5. **关键字优先与提前启动**：加载缓存时不是等完整加载完就唤醒CPU，而是边加载，同时CPU也拿到已经加载好的数据了
        
        1. 关键字优先：发出缺失字的请求，再处理cache行中的其他字
            
        2. 提前重启：缺失的字返回时马上给到流水线
            
    6. **合并写缓存**：写块不是马上执行，后面再对改地址写入，则直接写到缓存，实现对次写入合并为一次写入，提高效率
        
    7. Compiler Optimizations to Reduce Miss Rate: 按行优先/列优先数组在内存中的顺序访问数组元素。如算法不允许如此（例如矩阵乘法），也可以用分块的方式减少Cache缺失。
        
    8. **硬件预取**：提前把可能用到的块加载到缓存，而不是出现未命中再处理
        
    9. **软件预取**：编译器在代码中提前添加加载缓存的命令，降低缺失
        
    10. HBM
        
6. 关于组相联的进一步解释：组相连将cache 内分为多个组，每个组内有多个块，这些块地址符合某种规律（例如 块地址%块大小 == 特定值）。需要替换块时只能在组内替换。当只有一个组时，此时就是全相联。当组内只有一个块时，就是直接映射。
    
7. 当然，除了速度方面的考虑，还需要对虚拟内存的支持（安全、虚拟化），这就涉及到TLB、用物理地址还是虚拟地址等细节
    
    1. 为了隔离用户跟内核线程，需要：
        
        1. 提供至少两种模式区分内核和用户
            
        2. 提供用户只读，内核可读写的接口
            
        3. 提供在用户态和内核态之间切换的机制
            
        4. 提供对内存访问的限制
            
    2. 为什么用虚拟机？
        
        1. 安全需求
            
        2. 操作系统在 安全和可靠性上表现不佳
            
        3. 数据中心、云上实现多个用户（虚拟计算机）共用一台物理计算机
            
        4. 处理器性能提高，虚拟化开销可接受
            
    3. 虚拟机的基本要求
        
        1. 不能与物理计算机的表现有差异
            
        2. 客户软件不能直接修改系统资源（需要通过虚拟机）
            
    4. 内存虚拟化支持：
        
        1. 内存分级：虚拟内存-物理内存-机器内存
            
        2. 影子页表
            
    5. 指令对虚拟化的支持

2. # 指令级并行 ILP
    

## 流水线基础

1. 基本的五级流水线
    
    1. 取指
        
    2. 译码（读寄存器）
        
    3. 执行
        
    4. 内存访问
        
    5. 写回寄存器
        
2. 流水线冒险
    
    1. 结构冒险：硬件资源冲突（例如：同时load数据和指令），通常需要插入停顿或者硬件资源冗余解决
        
    2. 数据冒险：前后指令存在数据依赖，前面的指令还没更新数据，下一个指令就开始访问，拿到了旧值，共有三种类型：
        
        1. RAW
            
        2. WAR
            
        3. WAW
            
        
            决绝办法：插入停顿、转发技术、代码重排
        
    3. 控制冒险：分支指令的下一条指令地址不确定。解决办法：分支预测、分支延迟槽（先往下走，如果失败需要清洗流水线）
        
3. 流水线异常
    
    1. 精确异常，保证异常前的指令都完成了，异常后的指令不受影响
        
    2. 不精确异常
        
    3. 精确异常实现复杂，早期以不精确异常为主
        
4. 常见指令在五级流水线中的运行
    

  

## 高级流水线

进一步提高性能：多发射技术。降低冲突，动态调度：记分板技术、Tomasulo算法。分支预测技术：相关分支预测器、竞赛预测器。

5. 为了实现一个周期内多个指令执行，需要将ALU的数量提高（多个整数运算单元，多个浮点运算单元），这样就一个一个时钟周期内发射多个指令到不同计算单元执行计算。但有前面提到的流水线冒险的三个问题这里也要克服：
    
    1. 结构冒险，相对好克服，只要对应硬件空闲时才发射对应指令
        
    2. 数据冒险，见后面介绍
        
    3. 控制冒险，见后面介绍
        
6. 指令级并行的两种方式：
    
    1. 静态，编译器实现，硬件简单
        
        1. 循环展开，减少 循环控制指令的开销
            
        2. 软件流水：循环不同迭代重叠执行（要求没有前后依赖）
            
        3. VLIW
            
    2. 动态，硬件实现，要克服前面提到的冒险问题（指令顺序发射，但是乱序执行）
        
        1. 记分板技术（现代CPU不使用）
            
        2. Tomasulo 算法：寄存器重命名、保留站等技术避免冒险
            
        3. 硬件推测，提高分支指令效率
            
        4. ID 级拆分为两级：
            
            1. 发射：译码，检查结构相关
                
            2. 读操作数：等待数据冲突解决，读取源操作数
                
    3. 动态调度的优点：
        
        1. 不用为每个架构编一个二进制文件（缺点也是架构设计需要兼容老版本）
            
        2. 可以解决编译时无法确定的依赖关系
            
        3. 解决无法预测的延迟，如cache缺失
            
7. 记分板技术：
    
    1. 硬件维护一个控制表，包括：
        
        1. 指令状态，记录每个指令执行到哪一步
            
        2. 功能单元状态，哥哥ALU是否空闲，正在执行什么操作，目标寄存器等
            
        3. 寄存器结果状态：哪些寄存器正在等待写入
            
    2. 硬件考虑上述信息，就可以确定接下来饿哪些指令可以发射执行，哪些指令还需要等待。从而避免冲突
        
    3. 记分板技术的缺点：
        
        1. 记分板的硬件可能成为性能瓶颈
            
        2. 不支持寄存器重命名，例如多条指令都使用同一个寄存器临时存储，且这些指令之间不依赖数据。但是这些指令需要等寄存器空闲才能依次执行。但实际只需要给他们换个寄存器就能并行。
            
8. Tomasulo 算法
    
      **指令****执行阶段**：Tomasulo 将指令执行分为三个阶段（与记分板略有不同）：
    
    2. 发射：
        
        1. 条件： 功能单元有空闲的保留站。
            
        2. 操作： 如果操作数已在寄存器中，则直接读取；如果尚未计算出来，则设置为等待该保留站标签。这一步本质上完成了寄存器重命名，消除了 WAR 和 WAW 冒险。
            
    3. 执行：
        
        1. 条件： 指令需要的所有操作数都已就绪（即不再等待任何保留站的结果）。
            
        2. 操作： 开始执行运算。执行时间因操作而异。
            
    4. 写结果：
        
        1. 操作： 计算结果和保留站标签被广播到公共数据总线上。
            
        2. 效果： 所有等待该标签的保留站和寄存器会立即捕获这个值。这使得依赖这条结果的指令可以马上开始执行。
            
    
      **Tomasulo 的优势**
    
    5. 寄存器重命名： 硬件动态地将寄存器重命名到保留站位置，彻底消除了 WAR 和 WAW 冒险。
        
    6. 分布式控制： 避免了记分板的集中式瓶颈。
        
    7. 精确中断： 配合 reorder buffer 可以支持精确异常。
        
    
      Tomasulo 算法是现代超标量处理器乱序执行引擎的理论基础。
    
      
    
9. 分支预测：控制冒险是ILP最大的障碍
    
    1. 基本预测器，两位饱和预测器：记录每个分支之前的状态，以此作为后续预测的依据
        
    2. 相关预测器：不仅考虑当前分支的历史预测结果，还同时维护一个全局的分支预测表，在预测表中查询其他相关分支的预测结果。该思想的原因是程序中经常有类似如下结构的分支代码
        
        ```Plain
        if (cond1)          // 分支 B1
            ... 
        if (cond2)          // 分支 B2
            ...
        if (cond1 && cond2)  // 分支 B3
            ...
        ```
        
    3. 竞赛预测器 **待补充**
        

10. # 数据级并行 DLP
    

数据级并行主要指同时对多个操作数进行运算。

1. 向量体系结构
    
    1. 向量体系结构将内存中分散的元素取到一个更大的寄存器堆中，对寄存器堆中的数据进行操作并将结果写回内存中。
        
    2. 有一个向量长度寄存器
        
2. SIMD 单指令多数据
    
    1. 没有向量长度寄存器，通过掩码来确定向量长度
        
    2. 主要是CPU扩展多媒体指令，对图像处理通常是非常多相同的数据进行简单的操作
        
    3. SIMD load数据到寄存器的带宽要求高
        
    4. SIMD 一般是系统库支持，在自己的程序中一般不直接使用，避免移植性问题
        
    5. 在通用CPU上 SIMD 的推广是失败的
        
    6. SIMD中使用的寄存器长度比较长，且可以拆分为多个宽度窄的寄存器
        
3. SIMT 单指令多线程（GPU 采用）
    
    1. GPU 的结构与SIMD 有些不同
        
    2. GPU 本质是一个IO 设备，因此GPU 更新指令集架构相对容易，不需要兼容老架构（GPU的调用需要通过驱动，不同GPU安装不同驱动就行，不需要更新实际的应用程序。但应用程序已经编译为CPU指令了，所以CPU 升级需要兼容老版本，否则老程序不能运行）
        
    3. 关于GPU的更多介绍后续补充
        

4. # 线程级并行 TLB
    

线程级并行具体分为两类，一类是同一个芯片上的多个核心的并行，另一个是是不同芯片之间互联，即多处理器

1. 为什么多处理器流行
    
    1. 继续增加硅面积来提高ILP性能收益很低，同时功耗也压不住
        
    2. 高端服务器需求旺盛
        
    3. 数据密集型应用在互联网的驱动下井喷
        
    4. 个人电脑提高性能的需求降低（图像除外）
        
    5. 有效运用多处理器架构的方法论逐渐成熟
        
    6. 多处理器降低芯片开发的成本
        
2. TLB 中的难题
    
    1. 程序中有限的并行性
        
    2. 通讯的高额开销
        
3. 并行编程模型
    
    1. 共享内存：所有处理器共享相同的物理地址空间
        
    2. 消息传递：每个处理器有私有内存，通过消息传递信息
        
    3. 数据并行：多个处理器对不同数据执行相同操作（没有缓存的问题）
        
4. 多核
    
    1. 由于功耗与散热问题，同一个硅片上不能有无限多核心（一般8～16个），同时核心越多，不同核心之间通信延迟会增加
        
    2. 每个核心一般有自己的L1 cache 共享更高层级的cache，例如共享L3 cache
        
    3. 核心访问自己的memory 很快，但访问其他核心的memory比较慢
        
5. 多处理器
    
    1. 多处理一般是同一个主板上的多个处理器，处理器之间的通信延迟更大
        
    2. 多处理器的结构扩展性更好
        

  

**缓存一致性** 无 论是多核还是多处理器，不同核心之间memory 的同步都是非常重要的，实现缓存一致性的两种方案是监听协议，目录协议

6. 监听协议
    
    1. 所有缓存控制器都监听总线上的事务
        
    2. 写缓存时先获取总线控制权，其他核心的缓存实效，更新后再重新加载
        
    3. 总线比较复杂，如何在单个时钟周期实现是非常有挑战性的，因此主要适用于多核心
        
7. 目录协议
    
    1. 使用目录记录每个内存被哪些核缓存
        
8. 同步元语
    
    1. 单周期实现原子的数据读写基本不可能。
        
    2. 替代方案是：使用一组指令实现，如果中间没有打断，指令成功，否则执行失败。
        
9. 内存模型
    
    1. 对程序员来说最好当然是没有冒险
        
    2. 放松 WAR ： TSO
        
    3. 放松 WAR WAW ：PSO
        

10. # 仓库级计算机 WSC
    

仓库级计算机主要关注成本、能效。

1. 仓库级计算机的散热已经成为不能忽略的问题
    
2. 仓库及计算机的电源效率也是不能忽略的成本，因此要尽可能提高电源效率。
    
    1. 除了电网的效率，CPU的能效也要考虑，往往可以降低一点频率，大幅降低功耗，同时性能不会下降太多。
        
    2. 往往会应用性能差一些的CPU
        
3. 仓库级计算机每个CPU的利用率往往仅10%，因此亚马逊等通过提供云计算服务售卖多余的算力
    
4. 最重要的是可靠性，硬件太多因此单个硬件的可靠性很高，但庞大的系统可能每天都有硬件出故障
    
    1. 使用自动软件管理，重启等回复软件故障的机器
        
    2. 使用自动软件管理，屏蔽掉损坏的硬件，集中人工修复
        
5. 仓库级计算机执行的都是相对简单的任务，例如web 搜索引擎算法等。与超算差别很大
