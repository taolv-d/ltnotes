# 虚拟化（CPU）

## 目标：

1. 充分利用CPU资源，当CPU等待IO时仍能执行其他任务

2. 简化编程模型，让每个进程都认为自己独享一个CPU

## 机制：

1. 硬件提供特权指令，只有操作系统能够执行，例如访问硬件IO、访问文件、定时器、中断等

2. 为了保证多个进程之间切换，操作系统在进程切换时保存 进程的栈、寄存器等信息，确保后续恢复进程运行

3. 为了实现目标1，进程访问IO等 会陷入操作系统，操作系统此时实现进程切换

4. 为了公平，操作系统注册定时器中断，中断时进行切换

## 进程调度：

1. 先来先服务

2. 最短任务优先

3. 轮转

4. 多级反馈队列

   1. 高优先级线运行

   2. 相同优先级轮班运行

   3. 新进程分配最高优先级

   4. 每个进程有最大时间片，时间片用完，降低优先级

   5. 经过一段时间后，所有任务都提高为最高优先级

优点：不需要先验知识，对于短时UI任务可以迅速响应，长时间CPU任务也能正常推进

* 彩票算法

## 多CPU调度：

1. 单队列调度：从单个CPU的调度方式修改而来，所有核心共享一个调度队列。

   1. 问题：同步开销（调度队列）、缓存亲和性，如何尽可能让一个CPU执行特性进程

2. 多队列调度：每个CPU核心有自己的调度队列：

   1. 能够解决单队列的问题

   2. 但是 容易出现负载不均匀问题

## 方案

1. 简单虚拟内存：利用硬件实现基地址+偏移的方式实现虚拟内存。但由于堆跟栈之间未使用的空间很大，造成内存浪费。此外还容易出现碎片化

2. 分段：将程序分段，分为程序代码段、栈段、堆段等，每个段都有自己的基地址+偏移，实现节约内存。但无法避免碎片化。此外还可以实现代码段复用

3. 分页：将内存划分为固定大小的页，页为最小单位。这样避免了内存碎片问题（大家都一样大，不存在碎片）。但是 页表通常很大（映射完整地址空间），此外地址转换会更慢

4. 快速地址转换 TLB：为了解决页表比较大、比较慢的问题，硬件提供TLB 实现页表的缓存，TLB命中，可以快速转换到物理内存。但TLB大小有限制，未命中仍会很慢

5. 多级页表：

   1. **目的：**&#x4E00;级页表非常大（4G内存，页大小4K，有2^20个页表项，每个页表项4字节（32位系统对齐），页表大小为4MB），每个进程都需要一个页表，因此几百个进程光页表就要几百兆

   2. 方案：

      1. 更大页表，空间浪费

      2. 分页+分段（只为使用的段提供页表）。大而稀疏的堆可能造成大量的页浪费

      3. 多级页表：一级页表下面是多个二级页表，（甚至是3级页表）。最后一级页表指向物理内存。一级页表为空，则不需要记录其子页表，节省内存。（八叉树类似的结构）

      4. 反向页表：页表只记录分配了物理内存的页表（这样所有进程的页表加起来最大才4MB）。问题是需要线性查找页表，或者使用散列表

## 如何提供巨大的虚拟内存（物理内存受限）：

1. 交换空间，将长期不用的内存放入磁盘

2. 存在位：表示页表在内存还是在磁盘

3. 页错误：页不在内存时，发生页错误，操作系统加载磁盘上的页，并重新加载页

4. 内存满了：内存有高水位线，确保有空闲内存可用。超过水位线时，操作系统选择不常用的页踢到内存

## 内存交换策略：

1. 最优替换（理想模型，无法实现，无法预知未来，但是可以作为参考）：将最后用到的页换入磁盘

2. 先入先出 / 随机，可能会提出一个重要的页，该页马上就要用

3. 最近最少使用（LRU）：完整实现需要频繁扫描所有页，因此开销较大

4. LRU 近似实现：时钟算法：每个页都有一个使用位，时钟循环依次扫描每个页，并将扫描过页的使用位置0。如果本轮扫描时，对应页的使用位有效，说明在一个时钟周期内，该页被用了，不应该踢出。如果无效，则没有用，可以踢掉。

   1. 最差情况也只需要一次遍历所有页

   2. 考虑脏页（被修改过的页，需要同步到硬盘），踢出脏页需要同步磁盘，同时脏页后续可能还会被写入，因此踢出脏页成本更高，因此优先踢出干净页。

   3. 内存抖动：内存不够，系统连续不断踢出页，同时从交换区加载。解决方案：加内存。或者干掉一个内存密集型进程（Linux）

# 并发

难点：共享数据、死锁

## 锁：

1. 临界区关闭中断（早期，单处理器）：中断交给用户程序风险太高。多核处理器不支持。丢失中断，导致严重系统问题

2. 硬件提供的原子指令：

   1. 测试并设置：返回旧值，同时原子的设置新值。

      1. 有锁1，无锁0

      2. 加锁时：设置为1，并检查返回值是否=0；无论是否拿到锁，锁都是1

      3. 解锁时：设置0

      4. 可以实现自旋锁

   2. 比较并交换：对于3个值 ptr expected new, 如果 ptr == excepted, 则ptr=new. 返回ptr最开始保存的值（交换前的原始值）

      1. 有锁1， 无锁0

      2. 加锁时：尝试交换为1，即 excepted=0， new = 1。返回0 则加锁成功

      3. 解锁时： excepted=1， new=0， 返回1 解锁成功

   3. 链接的加载和条件式存储

      1. 链接的加载：从内存去值到寄存器

      2. 条件式存储：上一次加载的地址在期间没有更新时才会成功

      3. flag：有锁1， 无锁0

      4. 加锁1：首先不断读取锁flag，只有为0时才能加锁（使用链接的加载）

      5. 加锁2: 一旦锁flag为0（即无人占用），此时调用条件存储写入flag = 1。注意这里可能会因为中断、上下问切换等，造成加锁1 加锁2 之间其他进程占用了锁，即锁flag以及为1.

      6. 加锁3: 价差条件存储返回值，成功则拿到锁（中途没有其他线程插入），失败，被其他线程抢走了锁

      7. 解锁：flag 置 0

   4. 获取并增加：

      1. 每个线程有一个自己的序号

      2. 锁相当于一个index

      3. index与线程的序号相等时，该线程执行

      4. 线程执行完后解锁，解锁对index+1，则轮到下一个线程运行

## **条件变量**

线程需要检查某一条件（condition）满足之后，才会继续运行

条件变量通常需要内核或硬件支持。

### 以下是条件变量的典型使用步骤

* **定义条件变量和互斥锁**：

std::condition\_variable cv;

std::mutex mtx;

* **生产者线程**： 生产者线程在修改共享数据后，调用 *notify\_one()* 或 *notify\_all()* 通知等待的线程。

std::unique\_lock\<std::mutex> lock(mtx);

shared\_data = new\_value; // 修改共享数据

cv.notify\_one(); // 通知一个等待线程

* **消费者线程**： 消费者线程在访问共享数据前，使用 *wait()* 等待条件满足。

std::unique\_lock\<std::mutex> lock(mtx);

## 信号量

1. sem\_wait() 信号量-1，如果为负则等待，否则继续执行

2. sem\_post() 信号量+1，如果有等待线程，就唤醒一个

二值信号量（行为类似锁，但不能替代锁）：信号量的初值给 1

1. 加锁时信号量-1，变为0，继续运行

2. 其他线程在加锁时，信号量-1为负，阻塞

3. 解锁时信号量+1为0，唤醒阻塞线程

条件变量（行为类似条件变量，但不能替代条件变量）：信号量初值给0

1. 等待条件变量的信号加锁-1为负，阻塞

2. 唤醒线程给信号量+1，唤醒一个等待线程

信号量可以通过锁+条件变量实现：锁用于保护计数器，条件变量用于唤醒阻塞的线程

使用信号量实现锁或条件变量是很困难的：

1. 对于锁：

   1. 锁只能在同一个线程内解锁（其他线程解锁会失败或未定义行为），信号量可以在其他线程post

   2. 锁可以重入，即同一线程内可以再次对同一个锁加锁。但是信号量加锁两次可能死锁

2. 条件变量

   1. 条件变量需要原子的释放锁+休眠，使用信号量无法实现原子的释放锁+休眠。如果中间post被调用就会造成丢信号，将无法唤醒。

   2. 因此条件变量需要内核或者硬件支持

## 并发常见问题

1. 非死锁缺陷

   1. 违反原子性：常见例子就是判空跟使用中间 该值可能被其他线程置空

   2. 违反顺序缺陷：线程1初始化，线程2使用，实际可能2先执行，但1还没起来

2. 死锁缺陷：（由于代码封装，隐藏了内部对锁的需求）

   1. 死锁条件，及破解办法：

      1. 互斥：消除互斥条件，例如原子变量、无锁数据结构等

      2. 持有并等待： 加一个全局大锁（不好）

      3. 非抢占：trylock, 如果失败就释放所有资源

      4. 循环等待（环形依赖）：按顺序那锁释放锁

   2. 其他避免死锁的方案

      1. 调度，让可能产生死锁的线程在一个处理器上依次运行，消除了并行问题

      2. 死锁检测，适用极低概率死锁的系统，出现死锁直接重启

## RAID 廉价冗余磁盘阵列

1. RAID 0

   1. 条带化，一次写入分成多个条带，分别存入不同磁盘，最大并行度

   2. 没有冗余

2. RAID 1

   1. 镜像：容量一半

   2. 写入速度：将为带宽的一半，同一个数据写两份

   3. 读取锁度：实际也可能是一半，假设两块硬盘的状态完全一致，那么1号盘读文件的前半部分，2号盘读文件的后半部分，实际上2号盘需要先略过前半部分，定位到后半部分开始读，此时2号盘话费的寻转延迟时间跟从2号盘读完整文件是一样的（最差情况）。

   4. 最大限度的保护：最幸运情况可以损失一半的硬盘

3. RAID4

   1. 磁盘0～3 存条带数据，磁盘4存奇偶校验结果（异或）

   2. 最多可以挂一块，可以通过其他盘数据+奇偶校验恢复，若奇偶校验挂了，则重建奇偶校验即可

   3. 读取速度OK

   4. 写入速度0～3并行写

   5. 小写入问题：奇偶校验盘会成为瓶颈，如果随机修改0～3磁盘中的某些值（需要先读取奇偶校验值，更新后写入，其他硬盘只需要写入即可）

4. RAID5

   1. RAID4的改进，奇偶校验不是存储在一个硬盘中，而是有次序的分布在所有硬盘中。这样随机修改会发生在多个磁盘上，避免了瓶颈

## 文件系统的实现

1. 磁盘划分：

   1. 超级块（描述文件系统类型、inode 数量 inode起始位置）+ i bitmap(记录inode的空闲情况，使用位图)+dbitmap(记录data部分的空闲情况，也是位图)+inode+data

2. inode（index node）

   1. 描述了文件的信息：创建/修改时间、权限、拥有者、硬连接、分了那些块等等

3. 多级索引：

   1. inode通过记录那些块来找到文件，但块数量有限制。造成文件大小的限制

   2. 通过间接方法指向多个子inode实现大文件的支持

   3. 基于范围也能实现大文件支持，但不灵活，文件必须连续才行

4. 目录：

   1. 目录是inode编号与目录名或文件名的二元组。名字->inode号

5. 空闲空间管理：位图

6. 缓存和缓冲：

   1. 读取写入文件比较费时间。因此写入不会立即生效，而是等多个写入一次性完成，均摊成本

## 崩溃一致性 日志

如何保护文件系统：

文件系统更新 数据、inode、位图 时都可能发生崩溃

1. 使用全局检查程序，检查位图、inode、inode指针等是否有效，并修复操作系统（早期设备方案）

2. 预写日志(全数据日志)：

   1. 先将要写入的内容写入日志

   2. 将所有内容写入磁盘并加检查点

   3. 崩溃后先检查检查点，如果没有检查点，说明寄了，从日志恢复

3. 简化版本可以向日志写入校验信息（不能恢复数据），或者记录bitmap inode为脏

4. 元数据日志：

   1. 无日志情况：

      // 例如，创建文件需要多个步骤：

      1. 分配inode（索引节点）

      2. 初始化inode（设置权限、时间等）

      3) 在目录中添加条目

      4) 更新位图（标记inode已使用）

      // 如果系统在第3步后崩溃：

      * inode已分配但目录中没有条目 → 孤立inode

      * 或者相反：目录有条目但inode未分配 → 损坏的文件

   2. 元数据模式

      1. 把"要做什么"写入日志区域（原子操作）

      2. 标记日志为"已提交"

      3. 实际执行磁盘修改

      4. 标记日志为"完成"

   3. 元数据模式只记录了位图、inode、超级块、目录等，因此开销小

5. 写时复制

   1. 所有的更新全都写入空的区域，不回覆盖旧数据，哪怕崩溃，仍然能恢复到旧版本

   2. 写成功后在回溯清理旧版本
